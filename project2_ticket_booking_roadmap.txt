PHASE 3 — BOOKING MODULE (CORE — most important phase)
----------------------------------------------------------------
Goal: Booking works correctly under concurrency. Only one user
      can book the same seat. Payment failure rolls back cleanly.
Done when: Concurrency test passes (see below).

Tasks:

  [ ] Create Booking entity and repository
  [ ] Create PaymentService (simulation only — no external API)
        - Math.random() > 0.3 = success, else failure
        - This is intentionally simple
  [ ] Create BookingService with this exact flow:

        @Transactional
        public Booking createBooking(Long userId, Long seatId) {
          // 1. Fetch seat with PESSIMISTIC_WRITE lock
          Seat seat = seatRepository.findByIdWithLock(seatId);
          // lock held until transaction ends

          // 2. Check if already booked
          if (seat.getStatus() == BOOKED) {
            throw new SeatAlreadyBookedException();
          }

          // 3. Simulate payment
          boolean paid = paymentService.process();

          if (paid) {
            // 4a. Mark seat booked, save booking CONFIRMED
            seat.setStatus(BOOKED);
            seatRepository.save(seat);
            return bookingRepository.save(
              new Booking(userId, seatId, CONFIRMED, SUCCESS));
            // transaction commits here — lock released
          } else {
            // 4b. Save CANCELLED booking, throw to rollback
            bookingRepository.save(
              new Booking(userId, seatId, CANCELLED, FAILED));
            throw new PaymentFailedException();
            // transaction rolls back — seat stays AVAILABLE
          }
        }

  [ ] Add @Lock(LockModeType.PESSIMISTIC_WRITE) to seat query
      in SeatRepository
  [ ] POST /bookings endpoint (USER role only)
  [ ] GET /bookings/my — user sees their own bookings

  CONCURRENCY TEST (mandatory — do not skip):
  [ ] Write ConcurrencyTest using ExecutorService + CountDownLatch
        - Create 1 event, 1 seat
        - Submit 8 booking requests simultaneously
        - After all threads finish, assert:
            bookingRepository.countByStatusAndSeatId(
              CONFIRMED, seatId) == 1
        - Test must pass consistently (run 3 times)

  Failure scenarios to verify manually:
  [ ] Two users race → only one gets CONFIRMED
  [ ] Payment fails → seat stays AVAILABLE, booking is CANCELLED
  [ ] Booking a BOOKED seat → SeatAlreadyBookedException thrown

PHASE 4 — REDIS SOFT LOCK (add only after Phase 3 is stable)
----------------------------------------------------------------
Goal: Redis acts as an optimization layer. DB is still source
      of truth. If Redis is down, system still works.
Done when: Hold endpoint works, fallback works when Redis off.

Tasks:
  [ ] Add POST /seats/hold endpoint
  [ ] HoldService logic:
        - SET seat_lock:{eventId}:{seatId} userId NX EX 300
        - If key exists → return 409 Conflict (seat already held)
        - If set succeeds → return 200 OK (hold granted, 5 min)
  [ ] Release lock after booking completes (success or failure)
  [ ] Graceful fallback — wrap Redis calls in try/catch
        - If Redis throws → log warning, allow request through
        - DB pessimistic lock handles concurrency in fallback
  [ ] Test: stop Redis container, verify booking still works

Important rule:
  Redis soft lock = user experience optimization (prevents
  wasted DB lock waits). It is NOT a replacement for the
  DB pessimistic lock. Both must exist. DB lock is the
  guarantee. Redis lock is the courtesy.
